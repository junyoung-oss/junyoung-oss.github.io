# 🐞 결함 유형 (Defect Types)

> 테스트를 하다 보면 다양한 형태의 결함을 만나게 된다.  
> 결함을 유형별로 정리해두면 보고와 분석이 훨씬 명확해진다.

---

## 📌 결함이란?

요구사항, 설계, 코드, 문서 등에서  
기대 결과와 실제 결과가 다른 상태를 말한다.

---

# 🎯 주요 결함 유형

## 1️⃣ 기능 결함 (Functional Defect)

기능이 요구사항과 다르게 동작하는 경우.

예)
- 로그인 버튼 클릭 시 아무 반응 없음
- 할인율 계산 오류
- 필수값 미입력인데 저장 가능

👉 가장 기본적이고 많이 발견되는 유형

---

## 2️⃣ UI/UX 결함

화면 표시나 사용자 경험 관련 문제.

예)
- 버튼 정렬 깨짐
- 텍스트 오탈자
- 반응형 깨짐
- 다크모드에서 글자 안 보임

👉 사용성에 직접적인 영향

---

## 3️⃣ 성능 결함 (Performance Defect)

속도, 처리량, 응답시간 관련 문제.

예)
- 페이지 로딩 10초 이상 지연
- 대량 데이터 조회 시 멈춤
- 트래픽 증가 시 서버 다운

---

## 4️⃣ 호환성 결함 (Compatibility Defect)

환경에 따라 다르게 동작하는 경우.

예)
- 크롬에서는 정상 / 사파리에서 오류
- iOS에서 버튼 클릭 안됨
- 특정 해상도에서 레이아웃 깨짐

---

## 5️⃣ 보안 결함 (Security Defect)

보안 취약점 관련 문제.

예)
- 인증 없이 접근 가능
- SQL Injection 가능
- 개인정보 노출

👉 심각도(Severity)가 높은 경우가 많음

---

## 6️⃣ 데이터 결함 (Data Defect)

데이터 처리, 저장, 표시 오류.

예)
- DB 저장값 불일치
- 날짜 계산 오류
- 통계 수치 다름

---

## 7️⃣ 문서 결함 (Documentation Defect)

기획서, 요구사항 문서 오류.

예)
- 화면 기획서 오타
- 잘못된 필드 설명
- 요구사항 누락

👉 개발 전에 발견하면 가장 이상적

---

# 🔥 심각도 기준 예시

- **Critical** : 서비스 중단, 결제 불가
- **Major** : 핵심 기능 오류
- **Minor** : UI 깨짐, 오탈자
- **Trivial** : 단순 문구 수정

(※ 참고로 이부분은 회사마다 기준은 다를 수 있음)

---

# 🧠 QA 관점에서 중요한 점

- 결함은 "많이 찾는 것"이 목표가 아니다.
- "중요한 결함을 빨리 찾는 것"이 핵심이다.
- 유형을 구분하면 재발 방지 분석에 도움이 된다.

---

# 📌 내가 정리한 핵심

결함은 단순한 버그가 아니라  
품질 개선을 위한 데이터다.

---

> 좋은 QA는  
> 결함을 분류하고, 분석하고, 재발을 막는다.
