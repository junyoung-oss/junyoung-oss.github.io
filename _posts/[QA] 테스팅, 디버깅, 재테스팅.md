# 🏗️ 개발 단계별 결함

> 결함은 테스트 단계에서 생기는 것이 아니다.  
> 대부분은 개발 이전 단계에서 이미 시작된다.

---

# 📌 왜 단계별로 나눠야 할까?

> 결함은 소스 코드를 포함해 최종적으로 소프트웨어 동작의 장애를 유발 할 수 있는 모든 개발 산출믈에 존재할 수 있기 때문이다.

- 결함의 원인을 추적하기 위해
- 재발 방지를 위해
- 프로세스 개선 포인트를 찾기 위해

👉 결함은 "결과"가 아니라 "과정의 산물"이다.

---

# 1️⃣ 요구사항 단계 결함

기획/요구사항 정의 단계에서 발생하는 문제.

📌 특징
- 모호한 표현
- 누락된 조건
- 상충되는 요구사항

예)
- "빠르게 처리" 같은 정량화되지 않은 문장
- 예외 처리 조건 미정의
- 모바일 조건 누락

👉 이 단계에서 발견하면 비용이 가장 낮다.

---

# 2️⃣ 설계 단계 결함

아키텍처, DB 설계, 화면 설계 등에서 발생.

📌 특징
- 구조적 문제
- 확장성 고려 부족
- 데이터 설계 오류

예)
- 인덱스 미설계로 성능 저하
- 상태값 정의 누락
- 화면 흐름 충돌

👉 나중에 수정하면 영향도가 큼

---

# 3️⃣ 구현(코딩) 단계 결함

개발자가 코드를 작성하면서 발생.

📌 특징
- 로직 오류
- 조건문 실수
- 예외 처리 누락

예)
- if 조건 반대로 작성
- Null 처리 누락
- 계산식 오류

👉 QA가 가장 많이 발견하는 유형

---

# 4️⃣ 통합 단계 결함

모듈 간 연동 과정에서 발생.

📌 특징
- API 연동 오류
- 데이터 매핑 오류
- 환경 설정 문제

예)
- 응답 코드 다름
- 필드명 불일치
- 타임존 차이

👉 단위 테스트에서는 발견되지 않는 경우 많음

---

# 5️⃣ 운영 단계 결함

실제 서비스 환경에서 발생.

📌 특징
- 트래픽 이슈
- 환경 차이
- 예외 케이스 노출

예)
- 특정 브라우저에서 오류
- 대량 데이터 처리 시 장애
- 실서버 설정 누락

👉 가장 비용이 많이 드는 단계

---

# 💸 결함 수정 비용의 증가 (Cost of Defect)

> 결함은 늦게 발견될수록 비싸다.

결함이 발생했을 때, 해당 단계에서 적절하게 검출하고 제거하지 않으면  
이후 단계를 거치며 소스코드와 산출물 전반에 영향을 미치게 되고  
결국에는 실제 서비스 장애로 이어질 수 있다.

만약 결함이 수정되지 않은 상태로 다음 단계로 전달되면,  
그 결함을 제거하기 위해 더 많은 시간과 비용이 발생한다.

---

## 📊 단계별 결함 수정 비용 예시

| 단계 | 상대 비용 |
|------|------------|
| 요구분석 단계 | 0.1 ~ 0.2 |
| 설계 단계 | 0.5 |
| 코딩 단계 | 1 |
| 인수 테스트 단계 | 5 |
| 유지보수 단계 | 20 |

※ 위 수치는 절대값이 아니라 "상대적 비용 증가"를 의미한다.

---

## 🔍 왜 비용이 증가할까?

### 1️⃣ 영향 범위 확대
- 이미 여러 기능과 연결되어 있음
- 수정 시 다른 기능에 영향 가능

### 2️⃣ 재작업 증가
- 설계 수정 → 코드 수정 → 테스트 재수행
- 문서 및 배포 프로세스 재진행

### 3️⃣ 신뢰도 하락
- 운영 중 장애 발생 시 고객 신뢰 감소
- 브랜드 이미지 손상

---

# 🧠 QA 관점에서 중요한 점

- QA는 단순히 "테스트 단계"의 역할이 아니다.
- 요구사항 리뷰부터 참여하면 결함을 조기에 차단할 수 있다.
- 좋은 QA는 결함을 찾는 사람이 아니라  
  **결함이 생기지 않도록 돕는 사람**이다.

---

# 📌 내가 이해한 핵심

결함은 갑자기 생기는 것이 아니라  
개발 생명주기(SDLC) 어딘가에서 시작된다.

---

> 가장 좋은 결함은  
> 테스트 단계 전에 발견된 결함이다.
